## §1. Executive Summary

### One-liner
**코드뉴비("코뉴")가 학습 중 막힌 순간, 200줄 이내의 Python/JavaScript 코드를 붙여 넣으면 — 픽셀 아트 인터페이스에서 숲(Forest)/나무(Tree)/잎(Leaf) 3계층으로 해설받고, 자동 태깅·아카이빙으로 학습 맥락이 누적되는 웹앱.**

### 핵심 차별화 3종 묶음
1. **로드맵 기반 길찾기** — 프론트엔드/백엔드/DB 전체 로드맵에서 "지금 내가 어디에 있는가" 시각화
2. **계층 시각화** — 숲/나무/잎 + 폴더 트리 병행 표현
3. **누적 맥락** — 자동 태깅 + 아카이빙으로 학습 흔적 영구 보존

### MVP 범위
- 언어: Python (P0) + JavaScript (P1, Closed Beta까지 추가)
- 입력 한도: 실질 코드 200줄 + 입력 토큰 4,000 토큰 이중 한도
- 출력 모델: Pre-MVP + Closed Beta = GPT-5 mini 단일 / Open Beta+ = 분리 정책
- 플랫폼: 웹 (PC 우선). 모바일은 v1.x

### 1차 배포 목표 (Pre-MVP, 옵션 A)
- 2026-05-22 (금)까지 본인 단독 사용 가능 상태로 출시
- 검증 시그널: 본인이 주 4-5회 이상 직접 사용

### 2.1 Business Goals
- **G-1**: 본인이 학습 도구로 일상화 (주 4-5회 이상, Pre-MVP 1주차)

## §4. User Flows (Pre-MVP 5종, 텍스트 정의만)

> 시각적 다이어그램은 Stage 3 (UX Flow) 산출물.

- **Flow-1 온보딩 [P0]**: 첫 접속 → 로그인 → 환영 → 레벨 4지선다 → 첫 분석 화면(예시 코드 미리 채움) → 30초 내 첫 결과
- **Flow-2 즉시 해설 (UC-1) [P0]**: 대시보드 → "새 분석" → 코드 붙여넣기 → 이중 한도 검증 → 언어 자동 감지 → "분석하기" → Forest 30초 내 → Tree → 모든 라인 짧은 해설 + 핵심 5개 깊은 해설 → 캐러필러 +1
- **Flow-3 누적 검색 (UC-2) [P0]**: 대시보드 → "내 분석 이력" 탭 → 검색창 → 태그+코드+Forest+Key Concepts+메모 OR 매칭 → 카드 리스트 → 상세 페이지
- **Flow-4 분석 결과 → 태깅 편집 → 아카이브 [P0]**: 분석 결과 → AI 자동 태그 → 인라인 편집 → 자동 저장 → 깊은 해설 핀(📌) 시 영구 저장
- **Flow-5 Leaf 추가 확장 [P0]**: 짧은 해설 영역 클릭 → 차감 안내 모달 → "예, 깊게 보기" → 깊은 해설 확장 (5개당 분석 1회 차감) → 일일 한도 0 시 단순 차단

#### ✏️ FR-ANALYSIS-006 [P0] Leaf 통합 생성 + 입력 전처리

#### ✏️ FR-ANALYSIS-007 [P0] 추가 Leaf 확장 (Leaf Expansion on Demand)
- **AC**:
  - GIVEN 분석 결과 화면, WHEN 사용자가 핵심 5개 외 라인 클릭, THEN 모달:
    ```
    🦜 이 라인 더 자세히 풀어볼까?
    깊은 해설 사용량: N / 5
    (5개 채우면 분석 1회 차감)
    [예, 깊게 보기]    [취소]
    ```
  - "예" 클릭 시 → **기존 짧은 1줄 해설을 깊은 해설로 확장** (대체 아님, 짧은 해설은 유지하면서 펼치기).
  - 5번째 확장 직전 → 일일 한도 -1 + 카운터 리셋 + "분석 1회 차감됐어 🦜" 알림.
  - 잔여 0회 시 차단 + 안내 (우회 옵션 없음, 자정 기준 갱신).
- **의존성**: FR-ANALYSIS-006
- **비고**: **5:1 차감 룰의 본질은 사용자 abuse 방지 페널티**. 추가 Leaf 1회 API 비용은 분석 1회의 약 1/15 수준이므로 운영자는 약간 이익. 정확한 비용 회수가 아닌 남용 방지 목적.

#### ✏️ FR-ANALYSIS-012 [P0] 학습자 레벨 적용 (시스템 프롬프트 변수 주입)

  - ☐ 레벨 1이면 모든 깊은 해설에 비유 필수 (FR-OUTPUT-003 AC와 일치).
  - ☐ 메뉴 4번(자동 추천)은 레벨값이 아님 — `learner_level_auto=TRUE`로 표현, 시스템이 1·2·3 중 하나 할당.

## §6. Non-Functional Requirements (NFR)

### §6.1 비용 5룰 (NFR-1 ~ NFR-5) — Discovery v4 §11 명문화

> **이 5룰의 본질**: GPT-5 mini API 비용이 사용자당 ₩400/월 흑자선을 넘지 않도록 강제하는 5중 안전장치. Pre-MVP에선 본인 단독이라 비용 무관하지만, 룰 자체는 **Closed Beta 진입 시 자동 발효**되도록 코드에 미리 박아둔다.

#### NFR-1. Prompt Caching 강제

- **수치**: 시스템 프롬프트(약 800 토큰) + Few-shot 예시(약 1,200 토큰) = 약 2,000 토큰을 OpenAI prompt caching에 항상 적재. 캐시 적중률 **≥ 90%** 목표.
- **검증**: API 응답의 `usage.prompt_cache_hit_tokens` 비율을 일배치로 측정.
- **연결 FR**: FR-ANALYSIS-005, FR-ANALYSIS-012
- **왜?**: 캐시 히트는 prompt 토큰 단가가 1/10. 시스템 프롬프트 변동 시 학습자 레벨 1·2·3 모두 캐시 분기되도록 prefix를 정렬.
- **비고**: 적중률 ≥90%는 Beta 정상 트래픽 기준(prefix가 hot 유지될 때). Pre-MVP 단독 사용 시 캐시 TTL 만료로 미달이 정상 — 무료 API라 무관, 비용 절감 주축은 NFR-3 SHA 캐시 (TRD §19.2).

#### NFR-2. 출력 max_tokens 8,000 강제 + 정밀 예산

- **수치**: API 호출 시 `max_tokens=8000` 고정 (옵션 A 확정, §6.5 참조). 응답이 잘릴 가능성 모니터링(`finish_reason == 'length'` 비율 < 1% 목표).
- **검증**: `finish_reason` 로깅. 1% 초과 시 알림 + 한도 8,500으로 임시 상향.
- **연결 FR**: FR-OUTPUT-004, FR-OUTPUT-003, FR-ANALYSIS-006
- **왜?**: 입력 4K + 출력 8K = 약 ₩14/회. Pre-MVP는 본인 단독이라 비용 무관, Closed Beta까지 안전 마진 확보. Open Beta+ 진입 후 비용 압박 시 §6.5 옵션 B/C 회귀 검토.
- **⚠ 산수 검증 결과는 §6.5 별도 표** (인계 문서 §3.8.1의 "5,800 추정"이 실제로는 약 7,550. §6.5에서 옵션 A로 한도 상향 확정.)

#### NFR-3. 동일 코드 1주 캐시

- **수치**: 사용자별로 코드 SHA-256 해시가 동일한 분석 요청은 **7일 이내** API 재호출 없이 DB 결과 반환. 캐시 적중 시 한도 차감 X, 캐러필러 보상 X.
- **검증**: `analysis_cache` 테이블 `hit_count` 누적, 월별 적중률 리포트.
- **연결 FR**: FR-ANALYSIS-011, FR-ARCHIVE-001
- **왜?**: 코뉴 페르소나는 같은 코드를 자주 다시 본다(복습 패턴). 적중률 30% 가정 시 월 ₩330 절감/사용자.

#### NFR-4. Leaf Lazy 저장 (Pre-MVP 부분 위반 명시)

- **원칙**: 추가 Leaf 깊은 해설은 **사용자가 📌 핀을 찍은 것만** DB 영구 저장. 미핀은 7일 분석 캐시 만료와 함께 휘발.
- **Pre-MVP 부분 위반**: FR-ANALYSIS-006이 분석 1회에 짧은 해설 + 핵심 5개 깊은 해설을 **통합 생성** 하므로, 이 5개는 자동 저장. NFR-4는 **추가 Leaf(6번째 이상)에만 적용**.
- **연결 FR**: FR-ARCHIVE-004, FR-ANALYSIS-007
- **왜?**: 안 보는 데이터로 DB 부풀리지 말 것. Open Beta+에서 lazy 회귀 검토 (인계 문서 §3.5).

#### NFR-5. 비용 모니터링 대시보드

- **수치**: 매일 자정 KST 배치 — 전일 API 호출 수·총 토큰·총 비용·사용자당 비용을 운영자 대시보드에 기록. 사용자당 평균 비용이 **₩400/월 초과 시 알림**, **₩600/월 초과 시 신규 가입 일시 차단**.
- **연결 FR**: FR-ARCHIVE-001 (분석 메타데이터 누적)
- **왜?**: Discovery v4 §11의 "예외 허용 한계비용 ₩55/명/월"의 약 7배가 안전 마진. 그 이상은 단가 재설계 필요.

### §6.3 보안 + 개인정보 (NFR-7, 통합)

> 「개인정보보호법」 + GDPR + 「전자금융거래법」 3중 충족. Pre-MVP에서도 반드시 켜둘 항목 우선.

#### 보안 (Security)

- **NFR-7.1 HTTPS 전용**: Vercel 자동 HTTPS, HTTP 접근 시 301 리다이렉트.
- **NFR-7.2 패스워드 해싱**: bcrypt cost factor 12. 평문 저장 절대 금지.
- **NFR-7.3 세션**: HTTP-only Cookie + Secure flag + SameSite=Lax. 30일 만료 (FR-AUTH-003).
- **NFR-7.4 SQL Injection**: 모든 쿼리 parameterized (SQLModel ORM 강제). raw SQL 금지.
- **NFR-7.5 XSS**: 사용자 입력은 DOMPurify로 sanitize. 검색 하이라이트의 `<mark>`만 화이트리스트 통과.
- **NFR-7.6 결제 빌링키**: AES-256-GCM 암호화 후 DB 저장. 복호화 키는 **Railway 환경변수**(백엔드)에 보관 — 빌링키 암복호화는 Railway 백엔드가 수행하므로 Vercel과 무관. 전용 KMS 도입은 Open Beta 하드닝 후보.
- **NFR-7.7 Rate Limiting**: IP당 분석 호출 분당 5회. 로그인 실패 5회 시 15분 잠금.

#### 개인정보 (Privacy)

- **NFR-7.8 코드 데이터의 성격**: 사용자 자산. OpenAI API는 입력 데이터를 기본적으로 모델 학습에 사용하지 않음(opt-in 미실시 시 기본 제외) — 별도 옵트아웃 스위치 불요. 남용 모니터링(abuse monitoring) 목적 최대 30일 보관 후 삭제. Zero Data Retention(ZDR)은 Open Beta 자체 OpenAI 계정 전환 시 검토. 처리방침·약관에 명시.
- **NFR-7.9 hard delete**: 계정 삭제 시 14일 유예 후 모든 PII + 분석 이력 hard delete (FR-SETTINGS-011).
- **NFR-7.10 데이터 이동권**: GDPR Art.20 — `.md`/`.json` 내보내기 (FR-ARCHIVE-010, FR-SETTINGS-010).
- **NFR-7.11 동의 기록**: 약관·개인정보처리방침 동의 시점·버전·IP를 별도 테이블에 영구 저장.
- **NFR-7.12 PG사 PCI-DSS 회피**: 카드번호 직접 보관 금지. 토스페이먼츠 빌링키만 보유.

#### 공급망 (Supply Chain)

- **NFR-7.13 공급망 통제**: 의존성·빌드 파이프라인 보안. **Pre-MVP부터** — lock 파일 커밋 + 버전 핀 고정(재현 가능 빌드 reproducible build), GitHub Dependabot 활성화(취약 의존성 자동 경보), GitHub 계정 2FA + 브랜치 보호. SBOM·의존성 서명 검증 등 무거운 통제는 Open Beta 하드닝 후보로 보류. (근거: `06-security.md` 부록 B — OWASP Top 10:2025 A03. 공급망 안건 옵션 A 확정.)

**왜 Pre-MVP에서도 풀 적용?**: 본인 단독이라 해킹 위험 작지만, **DB 스키마와 코드 패턴이 굳어버리기 때문**. Closed Beta에서 보안 추가는 코스트 5배.

### §6.4 AI 슬롭 차단 (NFR-8) — Discovery v4 §3 명문화

> AI가 만든 티 나는 디자인은 본 제품의 **차별화 절멸 요소**. 모든 디자인 산출물(Stage 4·8)에 negative prompt로 강제.

#### 절대 금지 (Hard Block)

- **배경색**: cream / off-white / beige / 따뜻한 회색 톤 → ❌
- **포인트색**: terracotta / burnt orange / dusty rose / 그라디언트 일체 → ❌
- **폰트**: Inter / Roboto / Arial / Helvetica / SF Pro → ❌
- **레이아웃**: 둥근 카드 + 부드러운 그림자 + 친근한 일러스트 조합 → ❌
- **스타일**: skeuomorphism, glassmorphism, neumorphism → ❌

#### 강제 적용 (Hard Force)

- **베이스 테마**: 다크 (`#1A1A1A` 계열) — FR-OUTPUT-009
- **아트 스타일**: 32×32 픽셀 + 4색 팔레트 (orange/yellow/green/black)
- **폰트 본문**: monospace **JetBrains Mono** 또는 **IBM Plex Mono** (FR-OUTPUT-008)
- **폰트 UI**: Pretendard (한글) 또는 Noto Sans KR
- **모서리**: 픽셀 보더 (계단형, anti-alias 끔)
- **애니메이션**: CSS `steps()` 함수 강제 (부드러운 ease 금지, 픽셀 끊김)

**왜?**: AI 코딩 학습자가 보는 화면이 또 AI 슬롭이면 **"AI한테 또 졌다"** 는 무의식 신호. 픽셀+다크는 게이밍 노스탤지어(NES·GameBoy 세대 + Z세대 retro 트렌드) 양쪽에 어필.

```json
]
```

**왜 마스터 테이블 안 쓰나?**: 정규화하면 Tag 마스터 + Analysis_Tag 조인 테이블 = 2개 추가. 분석 50건 미만에선 효용 0. 검색은 JSONB GIN 인덱스로 충분.

**Closed Beta 마이그레이션 계획**: Tag 마스터 분리 + 사용자별 태그 사용 빈도 집계 + 태그 자동 완성. 마이그레이션 스크립트는 TRD에서.

### §7.5 KeyConcept (도감 항목, User 관점 누적)

|필드|타입|제약|설명|
|---|---|---|---|
|`id`|UUID|PK||
|`user_id`|UUID|FK → User|사용자별 도감|
|`name`|VARCHAR(50)|NOT NULL|"재귀", "클로저" 등|
|`category`|ENUM||'data_structure' / 'algorithm' / 'library' / 'pattern' / 'domain'|
|`definition`|TEXT||LLM 생성 정의 (도감 페이지용, FR-GAME-009)|
|`first_seen_analysis_id`|UUID|FK → Analysis|처음 등장한 분석|
|`appearance_count`|INTEGER|DEFAULT 1|누적 등장 횟수|
|`created_at`|TIMESTAMPTZ|DEFAULT NOW()||

**비고 (`definition` 의미)**: `definition`은 *이번 분석 맥락에서 LLM이 생성한 정의*다. 같은 개념이 다른 분석에서 다른 정의를 가질 수 있다(LLM의 맥락 의존). 도감(`/encyclopedia`)에서는 `first_seen_analysis_id`가 가리키는 분석의 definition을 도감 표시용으로 사용하고, 이후 분석에서 등장하는 동일 개념의 그 시점 정의는 분석별 결과 컬렉션(LineExplanation·`key_concepts` JSONB) 안에 별도로 남는다.

**인덱스**: `idx_keyconcept_user_name` UNIQUE (user_id, name)

**비즈니스 룰 (FR-GAME-007)**:

- 분석 결과 Key Concepts에서 사용자에게 처음 등장하는 개념 → 새 row INSERT, 도감 슬롯 +1
- 기존 등장 개념 → `appearance_count` +1 UPDATE
- 도감 마일스톤(1·10·30개)은 뱃지 시스템(FR-GAME-008)과 연동

### §7.6 Reward (캐러필러 + 방패 + Streak 통합)

|필드|타입|제약|설명|
|---|---|---|---|

|`caterpillar_balance`|INTEGER|DEFAULT 0|🐛 잔고 (FR-GAME-001)|

**중요한 원자성**: 분석 완료 → `Analysis INSERT` + `Reward UPDATE` (caterpillar_balance, streak_current 등) **단일 트랜잭션** (FR-GAME-001). 실패 시 분석 한도 차감 롤백.

**왜 단일 테이블?**: Reward 관련 모든 상태(캐러필러·방패·Streak·칭호)는 **항상 함께 읽힌다**(헤더 표시) + **함께 갱신된다**(분석 완료 시). 정규화로 4개 테이블 분리 시 매번 JOIN. 단일 row는 lock contention도 적음.

## §8. Hard Constraints (절대 어기지 말 것)

> **§5 FR과 §6 NFR 위에 있는 메타 룰**. 이 10개를 어기는 변경은 PRD 갱신 없이 절대 불가. Stage 5 TRD·Stage 7 TDDoc·Stage 9 TDDev 모두 본 섹션을 SSoT로 참조.

|#|제약|수치/내용|위반 시 영향|
|---|---|---|---|
|HC-1|**입력 한도**|실질 코드 200줄 + 입력 토큰 4,000 (이중 한도)|비용 회수 모델 무너짐|
|HC-2|**출력 한도**|max_tokens **8,000** (NFR-2 갱신본)|회당 API 단가 ₩14 초과 시 사용자당 흑자선(₩400/월) 위협|
|HC-3|**3계층 구조**|Forest / Tree / Leaf 외 4번째 계층 추가 금지 (Stage 4 결정: 3계층 구조 유지, 세 번째 층 명칭만 가지→잎 변경)|제품 정체성 절멸|
|HC-4|**Leaf Tier 분리**|모든 실질 라인 짧은 1줄 + 핵심 5개 깊은 해설 자동 (FR-OUTPUT-003)|코뉴 의도 "line-by-line 보고 싶다" 배반|
|HC-5|**모델 단일성**|Pre-MVP·Closed Beta 모두 GPT-5 mini 단일 (분기 금지)|Open Beta+에서만 GPT-4o-mini 분기|
|HC-6|**AI 슬롭 차단**|NFR-8의 cream/terracotta/Inter/skeuomorphism 5종 금지|차별화 절멸|
|HC-7|**한계비용 ₩0 보상**|캐러필러·방패·도감·뱃지는 디지털 자산만 (예외: 출석왕 보상권 월 5회 GPT-5 mini, Open Beta+)|게이미피케이션 흑자 구조 붕괴|
|HC-8|**TDD 약자 분리**|Stage 7 = **TDDoc** (Technical Design Document) / Stage 9 = **TDDev** (Test-Driven Development). "TDD" 단독 사용 금지|9단계 워크플로우 혼선|
|HC-9|**결제 활성 시점**|사업자등록 완료 + 토스페이먼츠 가입 영업일 종료 후에만 결제 UI 활성. Pre-MVP·Closed Beta는 결제 카테고리 비활성 라벨|「전자상거래법」 위반|
|HC-10|**14일 hard delete 유예**|계정 삭제 신청 → 14일 유예 → 모든 PII 영구 삭제. 그 전 로그인 시 자동 복구|GDPR Art.17 + 개인정보보호법 제36조 위반|

**왜 10개로 압축?**: 11개 이상은 외워지지 않는다. 9단계 워크플로우 + 1개(TDD 분리)와 같은 자릿수로 둬서 인지부하 최소화.
